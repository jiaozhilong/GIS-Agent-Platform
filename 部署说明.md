# GIS-Agent Platform 本地部署与测试说明

面向本地一键部署与联调。涵盖后端（Spring Boot + PostgreSQL）、前端（Vite + React）、
Mock SSO / Mock IMA 联调，以及P8-1计费纵深的使用方式。

---

## 1. 环境要求

| 组件 | 版本 | 说明 |
|------|------|------|
| JDK  | 17+  | 后端基于 Spring Boot 3（沙箱实测 Java 20 可运行） |
| Node | 18+  | 前端 Vite 构建 |
| PostgreSQL | 14+ | 必须安装 **pgvector** 扩展（`schema.sql` 含 `CREATE EXTENSION vector`） |
| Maven | 3.8+ | 仅源码构建时需要（或用仓库内 `./mvnw`） |

> ⚠️ pgvector 是硬性依赖。Ubuntu 安装：`sudo apt-get install postgresql-14-pgvector`；
> 本地若未启用该扩展，后端启动会因 `CREATE EXTENSION IF NOT EXISTS vector` 失败。

---

## 2. 数据库准备

应用启动时 `spring.sql.init.mode=always` 会自动执行 `src/main/resources/db/schema.sql`
（幂等建表，重复启动安全）。只需准备好空库与账号：

```sql
-- 以 postgres 超级用户执行
CREATE USER gisagent WITH PASSWORD 'gisagent';
CREATE DATABASE gisagent OWNER gisagent;
-- 进入 gisagent 库后启用向量扩展（仅首次）
CREATE EXTENSION IF NOT EXISTS vector;
```

默认连接串：`jdbc:postgresql://localhost:5432/gisagent`，账号 `gisagent/gisagent`。
如需改连，用环境变量覆盖（见第 4 节）。

## 3. 后端启动

### 方式 A：直接运行 jar（推荐）

```bash
cd backend
./mvnw clean package -DskipTests        # 或 mvn -o package -DskipTests（已缓存依赖）
java -jar target/gis-agent-platform-0.1.0-SNAPSHOT.jar
```

### 方式 B：开发模式（热编译）

```bash
cd backend && ./mvnw spring-boot:run
```

后端默认监听 **8080**。前端 dev/preview 的 `/api` 代理默认也是 8080，开箱即可联调。

### 关键环境变量

| 变量 | 默认 | 说明 |
|------|------|------|
| `SERVER_PORT` | 8080 | HTTP 端口 |
| `DB_USERNAME` / `DB_PASSWORD` | gisagent / gisagent | 数据库账号 |
| `JWT_SECRET` | change-me... | JWT 签名密钥，**生产必须改长随机串** |
| `APP_MASTER_KEY` | 空 | API Key 加密主密钥（生产建议注入） |
| `IMA_MOCK_ENABLED` | true | IMA 知识库走 Mock（无需真实 IMA）；设为 `false` 启用真实连接器 |
| `IMA_OPENAPI_CLIENTID` | 空 | 真实 IMA 的 Client ID（来自 IMA 开放平台，**运行时环境变量注入，不落库/不进 UI**） |
| `IMA_OPENAPI_APIKEY` | 空 | 真实 IMA 的 API Key（同上来向，**仅显示一次，务必保存**） |
| `APP_SSO_MOCK` | false | 设为 `true` 启用内置 Mock IdP（无真实 IdP 联调） |
| `STORAGE_DIR` | ./data | 上传/导出文件目录（建议绝对路径） |
| `app.usage.input-price-per-1k` | 0.001 | 每 1K 输入 token 单价（元） |
| `app.usage.output-price-per-1k` | 0.002 | 每 1K 输出 token 单价（元） |

示例（Mock SSO + Mock IMA 纯本地联调）：

```bash
java -jar target/gis-agent-platform-0.1.0-SNAPSHOT.jar \
  --SERVER_PORT=8080 \
  --APP_SSO_MOCK=true \
  --IMA_MOCK_ENABLED=true
```

### 3.1 IMA 知识库对接（真实连通）

> **核心结论：知识库 ID 不是接入 IMA 的钥匙，真正的钥匙是后端启动时的
> `IMA_OPENAPI_CLIENTID` + `IMA_OPENAPI_APIKEY` 两个环境变量。** 界面上填的
> 「知识库 ID」只是平台内部标签，不会发往 IMA。

**① 取凭证**：浏览器打开 `https://ima.qq.com/agent-interface` → 获取 API Key
→ 跳转本地 IMA → 复制 **API Key**（仅显示一次！）与 **Client ID**。

**② 关闭 Mock 并注入凭证启动后端**（生产务必用环境变量，不要写进配置文件/代码）：

```bash
java -jar target/gis-agent-platform-0.1.0-SNAPSHOT.jar \
  --IMA_MOCK_ENABLED=false \
  --IMA_OPENAPI_CLIENTID=你的clientid \
  --IMA_OPENAPI_APIKEY=你的apikey
```

**③ 前端「IMA 知识库配置」页 → 添加**：`kbId` 随便起个标签（如 `kb-supermap-products`）、
`kbName` 展示名、`kbType` / `purpose` / 检索权重按业务填。

**④ 点「测试」**：此时走真实 `POST /openapi/note/v1/search_note_book`，凭证正确即返回连接成功。

**三个必要条件（缺一不可）**：

| 条件 | 缺了会怎样 |
|------|-----------|
| `IMA_MOCK_ENABLED=false` | 走 Mock 连接器，返回假数据，永远连不上真 IMA |
| `IMA_OPENAPI_CLIENTID` / `APIKEY` 已设 | 真实连接器 `search()` 直接返回空（`[IMA] 未配置 Client ID / API Key，跳过真实检索`） |
| 界面填了 kbId + 后端凭证齐 | 才能真检索；kbId 仅作结果归类标签 |

**当前设计两个注意点**：

- **`kbId` 不会隔离检索**：真实请求体只含 `search_type / query_info / start / end`，不含 kbId；
  IMA 检索的是「该套凭证对应账号下的全部笔记本」。若需「不同 kbId 搜不同知识库」，需改代码把
  notebook/知识库标识塞进请求体。
- **界面「测试」通过 ≠ 真连通**：Mock 模式下测试永远返回 `true`。真连通以
  `IMA_MOCK_ENABLED=false` + 凭证齐 + 后端日志出现 `[IMA]` 真实请求为准。

> 注：本平台当前 `RealIMAKnowledgeBaseConnector` 直接调用 IMA 裸接口 `search_note_book`，
> 未走 ima-skills 层。功能上可检索，但与「通过 API Key 调用 ima-skills 做流程编排」的既定路线不一致，
> 是否改为走 ima-skills 为后续架构决策。

## 4. 前端启动

```bash
cd frontend
npm install
npm run dev        # 开发服务器 http://localhost:5173，/api 代理到后端 8080
# 或生产构建 + 预览：
npm run build && npm run preview
```

- `vite.config.ts` 已配置 `server.proxy` 与 `preview.proxy`：`/api` → `http://localhost:8080`
  （可用 `VITE_API_TARGET` 覆盖代理目标）。
- `src/api/client.ts` 默认 `baseURL` 为相对路径 `/api`；若前端独立部署且需直连后端，
  构建前设置 `VITE_API_BASE=https://your-host/api`。

访问 `http://localhost:5173` → 登录页（城市三维场景 + 模型大爆炸动效）→ 进入工作台。

## 5. 登录与 Mock SSO

- **账号密码登录**：直接注册 / 登录。首个注册的普通用户为 `USER` 角色。
- **Mock SSO**（无需真实 IdP）：后端加 `--APP_SSO_MOCK=true` 后，登录页出现「SSO 登录」按钮。
  点击 → 跳内置 Mock IdP → 自动以 `sso.user@example.com` 建号（首次）并复用（二次）→ 重定向回前端带 `?sso_token=JWT`。
- **真实 IdP**：配置 `app.sso.*`（`authorize-url` / `token-url` / `user-info-url` / `redirect-uri` / `frontend-base`），
  并设 `APP_SSO_MOCK=false`。

> 提权为超管：本地测试可 `UPDATE users SET role='SUPER_ADMIN' WHERE username='你的账号';`
> 超管可见「用户与权限」「审计日志」「计费账单」等系统管理入口。

## 6. 计费纵深（P8-1）使用说明

计费纵深在「用量计费」之外新增 **组织月度配额管控 + 账期账单 + 超限告警**。

### 6.1 设置组织配额（仅超管）

- 侧边栏「计费账单」→ 选择组织 →「设置配额」：
  - **月度 Token 上限**：该组织每月 token 预算（合计 token）。
  - **告警阈值（%）**：当月用量达到预算的该比例即触发告警（默认 80%）。
- 对应接口：`PUT /api/billing/quota` `{ organizationId, tokenLimit, warnThreshold }`。

### 6.2 配额进度可视化

页面实时展示「本月已用 / 上限」进度条，达到阈值线（标记点）变橙红色提示。

### 6.3 超限告警（自动 + 手动）

- **自动**：每次流水线运行结束（`PipelineEngine.finishRun`）后，自动聚合该组织当月用量，
  超过阈值即写 **审计日志**（`action=BILLING_QUOTA_WARN`）+ 发 **站内通知**（`type=QUOTA`，
  通知对象 = 组织 ADMIN + 平台 SUPER_ADMIN）。同月去重，避免每次运行重复告警。
- **手动巡检**：超管可 `POST /api/billing/quota/check?orgId=1` 主动触发一次配额核算与告警
  （等价于运行结束时的自动检查，便于无 LLM 环境下验证）。

### 6.4 账期账单

- 超管点「按月生成账单」→ `POST /api/billing/invoices/generate?month=2026-07`（默认当月），
  聚合各组织当月所有 `pipeline_runs` 的 input/output/total token 与估算费用，生成 `DRAFT` 账单。
  幂等：重生成会覆盖该账期。
- 账单列表 `GET /api/billing/invoices`（超管不传 `orgId` 看全部，普通用户看本组织）。
- 单价沿用 `app.usage.input-price-per-1k` / `output-price-per-1k`。

## 7. 自动化验证脚本

仓库根目录提供纯标准库（无需 pip 安装）的集成验证脚本，使用 **8088** 端口：

```bash
# 1) 先以后端 8088 端口启动（脚本 BASE=http://localhost:8088）
java -jar backend/target/gis-agent-platform-0.1.0-SNAPSHOT.jar --SERVER_PORT=8088 &

# 2) 另开终端运行验证（需 PG 已就绪）
python3 p8_billing_verify.py      # 计费纵深：22 项断言（配额 CRUD / 告警 / 账单）
python3 p7_3_verify.py            # 用量计费聚合
python3 p7_2_verify.py            # SSO 链路
python3 p6_verify.py              # 用户/权限
```

脚本会自行注册测试账号、经 psql 种入带 token 的 `pipeline_runs`、断言接口与审计/通知落库，
结束后清理测试数据。

> 若后端跑在默认 8080，把脚本里的 `BASE = "http://localhost:8088"` 改为 8080 即可。

## 8. 常见坑

| 现象 | 原因 / 解决 |
|------|------|
| 启动报 `relation "xxx" does not exist` | `ddl-auto=validate`，表由 `schema.sql` 建；确认 PG 已起且库正确 |
| 启动报 `extension "vector" is not installed` | 未装 pgvector，见第 1 节 |
| 前端登录后接口 403 | 未带 JWT；检查 `localStorage.token` 与后端端口是否一致 |
| 流水线运行 FAILED | 沙箱/本地无外网或未配 LLM API Key；属预期，接入真实 Key 即有网络即可生成方案 |
| 配额告警不触发 | 该组织未设置配额；或当月已告警过（同月去重）；用「手动巡检」复核 |
| 账单 totalTokens 偏少 | 存量 `pipeline_runs` 的 token 列为 NULL（计费为新增能力，向前不回填），仅新跑的流水线带用量 |
