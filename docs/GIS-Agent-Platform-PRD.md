# GIS Agent 平台 — 产品需求文档 (PRD)

> **版本**: v1.2 | **日期**: 2026-07-16 | **作者**: 产品通  
> **产品信条**: PM 不是需求搬运工，是价值判断官。

---

## 1. 问题陈述

### 现状

GIS 行业解决方案工程师（以超图为例）的核心工作是：接收客户需求文档 → 做需求分析 → 匹配公司产品 → 形成行业解决方案 → 输出产品选型清单 + 技术交流 PPT。

当前这个流程中，**最耗时、最痛苦的环节是在各个 PPT / 产品文档中手动翻找资料，然后手工组装成方案**。虽然资料已迁移到 IMA 知识库，但检索和组装仍然是纯人工流程。

### 受影响的用户

- **主要用户**：GIS 解决方案工程师（乙方专业人士），他们懂 GIS、懂产品，但大量时间消耗在"搜资料 + 复制粘贴"的体力活上
- **间接用户**：销售/客户经理（需要快速出方案）、客户（期待更快的响应速度）

### 不解决的代价

- **效率损失**：方案工程师 60%+ 时间花在非创造性劳动上（搜 PPT、翻文档、手工组装）
- **方案质量波动**：不同工程师对产品理解深浅不一，方案一致性差
- **响应速度慢**：客户从提需求到看到方案的时间窗口长，竞品可能抢先
- **知识沉淀浪费**：公司有价值的产品知识库和历史案例，但没有形成可被 AI 高效利用的知识引擎

## 2. 目标

| # | 目标 | 衡量标准 |
|---|------|---------|
| G1 | 方案生成时间缩短 70% | 从接需求到出方案初稿的时间：从 1-2 天 → 2-4 小时 |
| G2 | 知识检索准确率 ≥ 85% | 产品匹配准确率（AI 推荐的产品中有多少被工程师采纳） |
| G3 | 方案覆盖度 ≥ 90% | 平台能处理常见 GIS 行业方案场景的比例（国土空间规划、智慧城市、自然资源、低空经济等） |
| G4 | 用户 NPS ≥ 40 | 方案工程师对平台的满意度评分 |
| G5 | 模板复用率 ≥ 60% | 用户使用预置模板（而非自定义拼装）完成任务的比例 |

## 3. 非目标 (Non-Goals)

| # | 不做什么 | 原因 |
|---|---------|------|
| NG1 | **不做甲方自助平台** | 目标用户是懂 GIS 的方案工程师，不是不懂技术的甲方客户。不需要"傻瓜式一问一答"界面 |
| NG2 | **不替代人工最终判断** | AI 输出的是高质量初稿，最终方案仍需工程师审核和微调。不追求 100% 无人干预 |
| NG3 | **不做企业级权限/角色管理** | 用户级 IMA/LLM 配置隔离已内置，不做 RBAC、部门级权限等企业级管理功能 |
| NG4 | **不做实时 GIS 空间计算** | V1 专注"知识检索 + 方案生成"，不涉及在线地图渲染、空间分析计算等重型 GIS 引擎 |
| NG5 | **不做移动端** | V1 仅 Web 端 |
| NG6 | **不做方案版本对比/Git 式管理** | 方案历史版本管理放到后续迭代 |

## 4. 用户故事

### 核心用户：解决方案工程师

**US-1：快速出方案（完整流程）**
> 作为方案工程师，我希望上传客户需求文档后，平台自动完成需求分析 → 案例推荐 → 产品匹配 → 竞品对比 → 架构图生成 → 方案框架 → 方案质检 → 方案输出 → PPT 输出的全流程，以便我能在 2 小时内拿到一份可直接用于技术交流的完整方案包。

**US-2：按需拼装工具（部分流程）**
> 作为方案工程师，我希望只选择我需要的工具（比如只需「需求分析 + 产品匹配」），跳过不需要的环节，以便灵活应对不同项目需求，不浪费时间跑用不上的步骤。

**US-3：使用预置模板**
> 作为方案工程师，我希望平台提供常见场景（国土空间规划、智慧城市、自然资源管理等）的标准流程模板，以便我不用每次都从头配置，选中模板直接跑。

**US-4：知识库自动感知更新**
> 作为方案工程师，我希望平台自动感知我连接的 IMA 知识库的内容更新，以便产品匹配和案例推荐始终基于最新信息。

**US-5：连接自己的 IMA 知识库**
> 作为方案工程师，我希望登录平台后连接自己订阅的 IMA 知识库，平台基于我的知识库做检索和方案生成，不同用户的知识库互相隔离，以便每个工程师使用自己熟悉的知识体系。

**US-6：自定义模型配置**
> 作为方案工程师，我希望在独立的「大模型配置」页面管理我自己的模型列表（添加 Provider、配置 API Key、测试连接），以便在工具执行时从我的模型列表中选择合适的大模型。

**US-7：产物可干预**
> 作为方案工程师，我希望能看到每个工具的中间输出（需求清单、产品选型建议），以便在发现偏差时及时调整，而不是等全部跑完才发现方向错了。

**US-8：架构图自动生成**
> 作为方案工程师，我希望平台根据产品选型和需求场景自动生成技术架构图，以便直接放入 Word 方案文档和 PPT，省去手动画图的时间。

### 扩展用户

**US-9：销售快速响应**
> 作为销售/客户经理，我希望用预置模板快速生成初步方案框架（产品选型 + 架构图），以便在客户沟通中第一时间展示专业能力。

**US-10：模板自定义**
> 作为资深方案工程师，我希望在预置模板基础上，自定义拖拽工具节点修改流程，以便处理特殊客户需求的复杂场景。

## 5. 需求

### 5.1 核心概念：工具超市 + 流水线组装台

平台的核心设计理念是 **"工具原子化 + 自由拼装"**：

- 每个方案生成环节（需求分析、产品匹配、架构图……）是独立的 **Tool**
- **每个 Tool 可以单独配置 Skill**（选择哪个知识库、哪个大模型、Prompt 模板）
- 用户通过拖拽工具节点串成**流水线（Pipeline）**
- 系统提供**预置模板**覆盖 80% 场景，高级用户可自定义拼装

### 5.2 9 件套工具箱

#### Tool-1：需求分析

| 维度 | 说明 |
|------|------|
| **输入** | 客户需求文档（PDF/Word/纯文本）+ 附件 |
| **处理逻辑** | LLM 读取文档 → 提取关键需求 → 结构化归类（功能需求/非功能需求/约束条件/行业场景） |
| **输出** | 结构化需求清单（JSON） |
| **可配置项** | 使用的 LLM Provider、Prompt 模板、输出 schema |
| **优先级** | **P0** — 整个流水线的起点 |

#### Tool-2：案例推荐

| 维度 | 说明 |
|------|------|
| **输入** | 需求清单 |
| **处理逻辑** | 在 IMA 历史案例库中检索相似项目 → LLM 综合排序推荐 |
| **输出** | 历史相似案例列表 + 参考价值评分 + 核心参考点 |
| **可配置项** | 检索知识库范围、相似度阈值、推荐数量 |
| **优先级** | **P0** — 提供方案参考基线 |

#### Tool-3：产品匹配

| 维度 | 说明 |
|------|------|
| **输入** | 需求清单 |
| **处理逻辑** | 在 SuperMap GIS 平台产品智答库中检索匹配产品 → LLM 综合推荐（含版本、匹配理由、覆盖度评分） |
| **输出** | 产品选型清单（表格）：推荐产品、版本、匹配理由、需求覆盖度 |
| **可配置项** | 检索知识库（产品库/智答库）、匹配策略权重 |
| **优先级** | **P0** — 核心价值环节 |

#### Tool-4：竞品对比

| 维度 | 说明 |
|------|------|
| **输入** | 产品选型清单 + 需求场景 |
| **处理逻辑** | 检索竞品信息 → LLM 逐项对比（功能/性能/生态/价格） |
| **输出** | 竞品差异分析表 |
| **可配置项** | 竞品知识库、对比维度 |
| **优先级** | **P1** — 增值分析，非必需但高频需求 |

#### Tool-5：架构图生成

| 维度 | 说明 |
|------|------|
| **输入** | 产品选型清单 + 需求场景 |
| **处理逻辑** | LLM 设计系统分层和组件关系 → 生成架构图（输出格式由 skill 配置决定） |
| **输出** | 技术架构图，可直接放入 Word 方案文档和 PPT |
| **可配置项** | 图片输出格式（SVG/PNG）、架构图风格（由 Prompt 和 skill 决定，不做平台级限定） |
| **优先级** | **P0** — 方案文档和 PPT 的关键组成部分 |

#### Tool-6：方案框架生成

| 维度 | 说明 |
|------|------|
| **输入** | 前面所有工具的产出（需求清单 + 产品清单 + 案例 + 竞品） |
| **处理逻辑** | LLM 综合所有中间产物 → 生成方案大纲 + 各章节要点 |
| **输出** | 方案大纲（Markdown） |
| **可配置项** | 方案结构模板、行业知识库 |
| **优先级** | **P0** |

#### Tool-7：方案质检

| 维度 | 说明 |
|------|------|
| **输入** | 完整方案草稿 |
| **处理逻辑** | LLM 逐维度审查（完整性/一致性/产品匹配合理性/竞品客观性） |
| **输出** | 质量评分 + 逐项改进建议 |
| **可配置项** | 质检维度、评分标准 |
| **优先级** | **P1** — 提升方案质量，V1 可先做基础版 |

#### Tool-8：整体方案输出

| 维度 | 说明 |
|------|------|
| **输入** | 所有前置工具产出 |
| **处理逻辑** | LLM 整合生成完整方案文档 |
| **输出** | 完整方案文档（Word .docx / Markdown） |
| **可配置项** | 文档模板、章节结构、封面样式 |
| **优先级** | **P0** |

#### Tool-9：PPT 输出

| 维度 | 说明 |
|------|------|
| **输入** | 方案文档 + 架构图 |
| **处理逻辑** | 按 PPT 模板生成幻灯片（含文字、架构图、对比表） |
| **输出** | PPT 文件（.pptx） |
| **可配置项** | **PPT 模板配置参数**（模板文件上传、品牌色、字体、Logo、封面布局、页码格式、架构图插入位置） |
| **优先级** | **P0** — 最终交付物 |

**PPT 模板配置参数详细说明**：

```
PPTTemplateConfig {
  templateFile: File             // 上传的 .pptx 模板文件（占位符定义幻灯片版式）
  brandColor: string             // 品牌主色（HEX，如 #1565C0）
  accentColor: string            // 强调色
  fontFamily: string             // 正文字体（如 "微软雅黑"）
  titleFontFamily: string        // 标题字体
  logoImage: File                // Logo 图片
  coverLayout: "left_aligned" | "centered" | "minimal"
  slideNumberFormat: string      // 页码格式（如 "{current}/{total}"）
  architectureDiagramPosition: "full_slide" | "right_half" | "bottom_third"
}
```

### 5.4 工具间数据流（Context Bus）

所有工具节点通过**共享上下文**传递数据，而非点对点耦合：

```
ctx = {
  "requirement_doc": "原始需求文档",
  "requirements": {     // Tool-1 写入
    "functional": [...],
    "non_functional": [...],
    "constraints": [...],
    "industry": "国土空间规划"
  },
  "cases": [...],        // Tool-2 写入
  "product_selection": [...], // Tool-3 写入
  "competitive_analysis": {...}, // Tool-4 写入
  "architecture_diagram": "svg/png",  // Tool-5 写入
  "solution_outline": {...},  // Tool-6 写入
  "quality_report": {...},    // Tool-7 写入
  "final_solution": "docx/md",  // Tool-8 写入
  "ppt": "pptx"            // Tool-9 写入
}
```

每个工具节点声明所需的输入 key 和输出 key，流水线引擎自动完成依赖解析。

### 5.5 LLM 可配置设计

大模型和 IMA 各自有**独立的配置页面**（「系统设置 → 大模型配置」和「系统设置 → IMA 配置」），用户在流程工具中使用时可自由选择使用哪个大模型。

#### 大模型配置页面

每个用户在 **「系统设置 → 大模型配置」** 页面管理自己的模型列表：

| 功能 | 说明 |
|------|------|
| 添加 Provider | 配置 Provider 名称、类型（OpenAI 兼容/本地）、API 地址、API Key（加密存储） |
| 模型列表管理 | 每个 Provider 下可添加多个模型（如 gpt-4o、deepseek-v3） |
| 连接测试 | 测试模型连通性和响应速度 |
| 默认模型设置 | 设置平台级默认模型 |

#### 模型 Provider 数据模型

```
ModelProvider {
  id: string
  userId: string                        // 归属用户
  name: string                          // e.g., "GPT-4o", "DeepSeek-V3", "本地 Ollama"
  type: "openai_compatible" | "local"
  endpoint: string                      // API 地址
  apiKey: string                        // 加密存储
  models: [                             // 该 Provider 下的可用模型
    {
      modelName: string
      contextWindow: number
      maxTokens: number
    }
  ]
}
```

#### 工具级模型选择

用户在配置工具节点时，从自己的模型列表中**选择使用哪个模型**来执行该工具：
- **使用的 Provider**：复杂推理（需求分析/产品匹配）→ GPT-4o；简单摘要/质检 → DeepSeek-V3
- **Temperature / MaxTokens**：创造性任务（方案生成）→ 高 temperature；确定性任务（产品匹配）→ 低 temperature
- **System Prompt**：每个工具预置默认 Prompt，支持用户自定义

### 5.6 IMA 知识库集成（用户级可配置）

**核心理念**：IMA 知识库不做平台级硬编码。每个用户登录后连接**自己的** IMA 订阅知识库，平台基于该用户的知识库做检索和方案生成。不同用户看到的是自己订阅的知识库内容。

#### IMA 配置（独立配置页面）

每个用户在 **「系统设置 → IMA 配置」** 页面管理自己的知识库连接：

```
UserIMAConfig {
  userId: string
  connectedKBs: [                          // 用户已连接的知识库列表
    {
      kbId: string                         // IMA 知识库 ID
      kbName: string                       // 知识库名称
      kbType: "subscribed" | "owned"      // 订阅的 / 自建的
      purpose: "product_doc" | "case_lib" | "industry_standard" | "competitor" | "general"
      searchWeight: float                  // 检索权重（0-1）
      enabled: boolean                     // 是否启用
    }
  ]
  autoSync: boolean                        // 是否开启自动感知更新
  syncInterval: number                     // 同步间隔（分钟）
}
```

#### IMA 配置页面功能

| 功能 | 说明 |
|------|------|
| 登录 IMA 账号授权 | 用户首次使用时授权平台访问自己的 IMA 订阅 |
| 知识库选择 | 从用户订阅的知识库列表中勾选要纳入平台检索的库 |
| 用途标注 | 标记每个知识库的用途（产品文档/案例库/行业标准/竞品），影响检索路由 |
| 权重调节 | 拖拽滑块调节不同知识库的检索权重 |
| 连接测试 | 测试知识库连接和检索是否正常 |

#### 检索路由

工具执行时，根据用户配置的知识库用途自动选择检索目标：
- **产品匹配（Tool-3）** → 自动路由到 `purpose=product_doc` 的知识库
- **案例推荐（Tool-2）** → 自动路由到 `purpose=case_lib` 的知识库
- **竞品对比（Tool-4）** → 自动路由到 `purpose=competitor` 的知识库

用户也可以在工具节点配置中手动覆盖检索范围。

#### 自动感知更新

- 通过 IMA MCP 接口定期轮询用户已连接知识库的变更
- 变更事件触发增量索引重建
- 检索结果标注"数据新鲜度"（最近更新时间）

```
│          │          │ 兼容协议)  │          │                  │
├──────────┴──────────┴──────────┴──────────┴──────────────────┤
│                     Context Bus (共享状态)                      │
├──────────────────────────────────────────────────────────────┤
│                     PostgreSQL 14 + pgvector                   │
│ (流程模板/项目记录/用户配置/向量检索)                            │
└──────────────────────────────────────────────────────────────┘
```

| 层 | 技术选型 | 理由 |
|---|---------|------|
| **前端** | React + TypeScript + Ant Design + React Flow | 既定技术栈；React Flow 原生支持可视化流程编排 |
| **后端** | Java Spring Boot 3.2 + Maven | 用户本地可运行调试；生态成熟、性能稳定 |
| **流程引擎** | 自研 DAG Scheduler（拓扑排序 + 状态机） | 轻量可控，Java `CompletableFuture` 做异步编排 |
| **模型接入** | OpenAI 兼容协议（Spring AI / 自封装 HTTP Client） | 统一接入 OpenAI / DeepSeek / 本地模型，标准 REST API |
| **知识库连接** | IMA MCP 协议（Java HTTP Client 对接） | 直接对接 IMA 知识库的搜索/列表/读取能力 |
| **数据库** | PostgreSQL 14 + pgvector（用户本地已安装 PG14） | 存储流程模板、项目记录、用户配置、向量检索增强 |
| **文件导出** | Apache POI（DOCX/PPTX） | Java 生态原生，成熟稳定 |

## 9. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 工具原子化 vs 全自动黑盒 | **工具原子化 + 自由拼装** | 方案工程师需要灵活性——有时只要产品选型、有时要全套方案。黑盒无法满足多变需求 |
| 预置模板 vs 纯自定义 | **模板为主(80%) + 自定义为辅(20%)** | 降低门槛，同时保留灵活性 |
| V1 人工审核 vs 全自动 | **V1 全自动通过** | MVP 追求速度，审核放在 Phase 3 |
| 后端语言 | **Java Spring Boot 3.2** | 用户本地可运行调试，生态成熟稳定；Maven 构建 |
| 数据库 | **PostgreSQL 14** | 用户本地已安装，生态成熟 |
| IMA 集成方式 | **用户级可配置** | 每个用户登录后连接自己的 IMA 订阅知识库，不做平台级硬编码 |
| LLM 集成方式 | **用户级可配置 + 独立配置页** | 大模型和 IMA 各有独立配置页面，工具执行时可选择使用哪个模型 |
| 架构图生成 | **LLM 生成，格式不限** | 格式由 skill 配置决定（SVG/PNG 等），不做平台级格式限定 |
| PPT 模板 | **可配置模板参数** | 支持上传模板文件 + 配置品牌色/字体/Logo/布局等参数 |
| 方案输出格式 | **Word .docx + Markdown** | Word 是行业交付标准，Markdown 便于在线预览和版本对比 |
| 项目管理 | **GitHub（新建公共仓库）** | 代码托管 + Issue 追踪 + Project 看板 + PR 评审 |
| 知识库更新感知 | **自动感知（增量索引）** | 知识库定期更新，手动触发会导致信息滞后 |
| 数据库 | **PostgreSQL 14 + pgvector** | 用户本地 PG14 + pgvector 向量插件做混合检索增强 |

## 10. 风险与对策

| 风险 | 影响 | 概率 | 对策 |
|------|------|------|------|
| IMA 检索质量不稳定 | 产品匹配/案例推荐不准确 | 中 | 支持 Rerank 二次排序；允许用户反馈标注 |
| LLM 生成的架构图不符合预期 | PPT 质量差，需要人工重画 | 中 | V1 直接输出 SVG，用户可在 PPT 中微调（draw 格式天然可编辑） |
| 超图官方知识库内容格式不统一 | 检索提取困难 | 高 | 在 Tool 层做内容预处理（PDF 解析、PPT 文本提取）；设置兜底策略 |
| 大模型幻觉（编造不存在的产品功能） | 误导方案工程师 | 中 | 强制要求每次产品匹配附带"知识来源引用"；方案质检节点可发现 |
| API 成本不可控 | 单个方案生成成本过高 | 低 | 支持模型分级：核心推理用强模型、简单任务用轻量模型 |
| IMA MCP 接口稳定性 | 平台依赖的外部服务不可用 | 低 | 检索结果缓存 + 降级策略（有缓存时用缓存，无缓存时走大模型自有知识） |
